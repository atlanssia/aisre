# AI RCA Workbench — PRD（产品需求文档，研发启动版）

> 文档目标：用于产品、研发、设计、测试、实施团队统一需求边界，支撑 MVP 开发与验收。
> 阶段：Phase 1（OO First MVP）

---

## 1. 产品概述

### 1.1 产品名称

**AI RCA Workbench**

### 1.2 产品定位

面向可观测平台（优先 OpenObserve）的智能根因分析工作台，聚焦：

- 告警智能分析
- 多信号证据压缩
- RCA 报告生成
- 人类决策支持
- 历史复盘沉淀

### 1.3 MVP 目标

在不替代 OpenObserve 的前提下，实现：

> 告警进入后 30~90 秒内自动生成结构化 RCA 报告。

核心收益：

- MTTR 显著下降
- 日志检索时间减少
- 一线值班门槛降低
- 复盘标准化

---

## 2. 用户角色

### 2.1 SRE / 运维值班

核心诉求：

- 快速判断是否真实故障
- 快速定位根因
- 获取下一步动作建议

### 2.2 应用研发

核心诉求：

- 快速关联 trace / 日志 / 最近变更
- 找到代码或依赖问题

### 2.3 技术负责人 / 管理者

核心诉求：

- 查看高危故障趋势
- 复盘历史 RCA
- 识别不稳定服务

### 2.4 平台管理员

核心诉求：

- 配置 OO Adapter
- 配置模型
- 管理 Prompt 模板

---

## 3. 用户故事（MVP）

### US-01 告警自动分析

**作为** 值班 SRE
**我希望** 告警触发后系统自动生成 RCA
**以便** 我无需手工翻日志即可判断根因。

### US-02 点击告警查看 RCA

**作为** 值班人员
**我希望** 点击告警后直接进入 AI RCA 报告页
**以便** 快速看到 TL;DR、证据链和建议。

### US-03 Drill-down 原始证据

**作为** 应用研发
**我希望** 点击关键日志/trace 可跳转 OpenObserve 原始页面
**以便** 做深度验证。

### US-04 历史相似故障

**作为** 技术负责人
**我希望** 查看类似历史 RCA
**以便** 快速判断是否重复问题。

### US-05 反馈 AI 结论

**作为** 值班人员
**我希望** 标记 AI 是否正确
**以便** 持续优化模型效果。

## 4. MVP 范围（P0）

### 4.1 必做范围

#### A. 告警接入

- OpenObserve Webhook
- 手工输入分析
- API 创建 incident

#### B. RCA 自动分析

#### C. RCA 报告页

- TL;DR

- 影响范围
- 建议动作
- 用户反馈

#### D. 历史报告

- 报告列表
- 报告搜索
- 服务过滤

#### E. 基础系统配置

- OO 连接配置
- LLM 配置
- Prompt 模板

### 4.2 暂不纳入 MVP

- 自动修复
- 多 Agent 编排
- 多后端（SigNoz）
- 复杂拓扑全图

- 值班排班

## 5. 核心业务流程

### 5.1 自动分析流程

```text
OO Alert → Webhook → Incident 创建 → Tool 检索 → LLM RCA → Report 持久化 → UI 展示
```

### 5.2 人工分析流程

```text
用户输入服务 + 时间范围 → 创建分析 → Tool 检索 → RCA → 报告
```

### 5.3 反馈闭环流程

```text
查看报告 → 标记正确性 → 写入 feedback → Prompt 优化
```

## 6. 页面级 PRD

### 6.1 页面A：首页 Alert Workbench

#### 页面目标

让值班人员 30 秒内完成：

- 判断严重性
- 查看 AI 是否已分析
- 进入 RCA

#### 页面模块

**1）高危事件列表**

字段：

- 严重级别
- 服务
- 告警摘要
- 首次发生时间
- AI 状态
- RCA 置信度

**2）Top 不稳定服务**

- 最近24h故障次数
- 平均恢复时长

**3）重复故障建议**

- 历史相似报告

#### 关键交互

- 点击卡片进入 RCA 报告
- 按服务筛选
- 按严重性筛选

---

### 6.2 页面B：RCA 报告页（核心）

#### 页面目标

让用户 1 分钟内完成：

- 理解根因
- 查看关键证据
- 确认建议动作

#### 模块1：TL;DR

字段：

- 根因结论
- 影响范围
- AI置信度
- 推荐动作摘要

#### 模块2：故障时间线

字段：

- 指标漂移
- 首条错误日志
- Trace 慢点
- Alert 触发

#### 模块3：关键证据

仅展示：

- 1条关键日志
- 1个关键 span
- 1个指标变化

#### 模块4：影响范围

- 根因服务
- 下游受影响服务

#### 模块5：建议动作

按：

- 立即止血
- 根本修复
- 长期治理

#### 模块6：反馈

按钮：

- 正确
- 部分正确
- 错误

---

### 6.3 页面C：历史 RCA 报告

支持：

- 时间范围
- 服务
- 根因类型
- 严重性
- 相似故障

## 7. 非功能需求

### 7.1 性能

- Webhook 到报告生成：≤ 90 秒
- 报告页打开：≤ 2 秒
- Drill-down：≤ 3 秒

### 7.2 可用性

- 服务可用性：99.9%
- Adapter 调用失败自动重试

### 7.3 安全

- Adapter Token 加密存储
- 报告访问 RBAC
- 审计日志记录

### 7.4 可扩展性

- Adapter 插件化
- Tool SDK 可新增工具
- LLM 可替换

---

## 8. 验收标准（UAT）

### 场景1：OO 告警自动 RCA

- 能成功接收 Webhook
- 90 秒内生成 RCA
- 报告包含根因、证据、建议

### 场景2：Drill-down

- 点击日志跳转 OO 对应查询页面
- 点击 trace 跳转 trace 详情

### 场景3：用户反馈

- 正确记录反馈
- 可在后台查看统计

### 场景4：历史检索

- 按服务和时间正确过滤
